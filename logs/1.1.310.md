# Roman Society 1.1.310

## 1.1.310 - House / Dynasty / Tree Coherence Refactor

This release reworks the membership model behind houses, dynasties, the family
trees and the Members screen so that all of them use a single, consistent
criterion. It also reverts the 1.1.308 change that had introduced floating live
branches.

### The single membership model now used everywhere

- **Real house member** = a non-slave character whose `resolveCharacterHouseId`
  resolves to that house (living or dead). This was already the de-facto source
  of truth; it is now the only one.
- **Real dynasty member** = a character whose resolved house belongs to the
  dynasty, checked directly through `houseBelongsToDynasty`
  (`gameDynastyIdForHouse(house) === dynastyId`). Because dynasty membership now
  derives from house membership, **every real house member is guaranteed to be a
  real dynasty member** (house tree is always a subset of the dynasty tree).
- **External spouse / guest** = the spouse of a shown member whose resolved house
  is not the current scope. Guests are shown but clearly labeled and never counted
  as real members.

### Fixed

- **Floating live branches / severed trees (regression from 1.1.308).** The
  1.1.308 "house ∩ dynasty" intersection removed members the (then buggy) dynasty
  set omitted, which cut parent-child links and turned children into isolated
  roots — producing disconnected groups, now with living characters. The
  intersection has been removed; coherence is instead guaranteed by the shared
  membership criterion above.

- **Characters in the house tree but not the dynasty tree (#5/#6).** Dynasty
  membership previously required `character.dynastyId === dynastyId` AND that the
  resolved house be present in `dynasty.houseIds` — a list that can be stale, so
  real members of a dynasty house were dropped from the dynasty tree. Dynasty
  membership now ignores that list and compares the house's dynasty id directly.

- **External spouses treated as real members (#3/#7).** `buildHouseTreeModel` used
  to add *any* spouse to the tree's allowed set, so a spouse from another house
  rendered as a normal member (no guest label). It now only promotes spouses who
  are themselves real members of the house; all other spouses render as labeled
  guests and never enter the member set.

- **Isolated ancestor / orphan blocks (#8).** A new connected-component pass
  (`keepConnectedFamilyComponents`) drops components that contain no real member
  (orphaned ancestors or stray spouses). It never hides a real member: any
  component containing a member or the focused character is always kept. For the
  dynasty tree this is a no-op (every node there is a real member).

- **Unexplained floating groups (#1).** When more than one disconnected line is
  shown (members that genuinely share no shown common ancestor), the extra lines
  now appear under a clear labeled divider ("Other known lines of this
  house/dynasty"), instead of as ambiguous side-by-side trees.

- **Members vs tree mismatch (#7).** The Members screen and the house tree now use
  the same real-member set (`buildHouseTreeModel` member set), so they no longer
  disagree about who belongs to a house. This is now backed by the coherent model
  above rather than being a stand-alone patch.

### Changed

- Runtime version aligned to `1.1.310` (also invalidates the per-version
  family-tree cache so trees recompute with the new rules).
- Added a styled section divider for additional family lines in the tree overlay.

### Notes / Not Confirmed

- Some real members can still appear as separate (now clearly labeled) lines when
  their connecting ancestors were never generated. This is a data/generation
  limitation, not a filtering bug; those members are kept visible by design rather
  than hidden. A deeper fix would link generated relatives to a common ancestor at
  generation time and is recommended as a follow-up.
- Verified statically (all modules parse); in-game verification of the visual tree
  is still recommended.
