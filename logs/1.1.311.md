# Roman Society 1.1.311

## 1.1.311 - One Family Graph For Both Tree Views

Previously the FULL DYNASTY TREE and the HOUSE FAMILY TREE were built by two
independent algorithms. They selected different ancestors, chose roots with
different rules, and ordered branches differently (the dynasty view even reordered
roots depending on the selected character). The result: the same people appeared
with a different structure and a different main/secondary line in each view.

This release makes both views share one graph, one root rule and one layout. The
only difference is scope.

### New unified model

- **One graph builder.** `dynastyTreeAllowedIdMap` builds the complete family graph
  of a dynasty (members of every house of the dynasty + the ancestor spine).
- **Dynasty view = the whole graph.**
- **House view = the same graph, filtered.** `restrictDynastyGraphToHouse` keeps the
  selected house's real members (the same set the Members screen uses, via
  `buildHouseTreeModel`) plus the ancestor spine taken from the dynasty graph, so a
  member keeps the *exact same* parent/child position it has in the dynasty view.
- **One root rule for both.** Both views now pick roots with
  `treeRootIdsFromAllowed(allowed, state, '', 24)` — note the empty focus: root
  selection and ordering are deterministic and **no longer depend on the selected
  character**. Focus is used only to highlight/centre the view.
- **Robust house list.** `dynastyHouseList` unions `dynasty.houseIds` with every
  house whose `gameDynastyId` matches, so the dynasty tree never omits a house even
  if the recorded list is stale.

### Fixed

- House tree and dynasty tree showing the same people with different structure,
  different main line, and different "Other known lines" ordering.
- A line appearing as the main line in one view and a secondary line in the other
  (caused by the house builder only following in-house ancestors, so a member whose
  connecting ancestor lived in another house became a root in the house view but a
  child in the dynasty view).
- Tree organisation changing depending on which character/menu the tree was opened
  from (focus-dependent root ordering removed).
- Dynasty tree potentially omitting a house when `dynasty.houseIds` was stale.

### Kept from previous releases

- Single membership criterion (`resolveCharacterHouseId` for houses;
  `houseBelongsToDynasty` for dynasties) from 1.1.310.
- External spouses render as labeled guests and never count as members.
- `keepConnectedFamilyComponents` still drops orphan components with no real member
  (no-op for the dynasty view, where every node is a real member).
- The labeled "Other known lines of this house/dynasty" divider for genuinely
  disconnected lines.

### Changed

- Runtime version aligned to `1.1.311` (invalidates the family-tree cache). The
  house-view cache key now also depends on the dynasty graph signature, since the
  house view is derived from it.

### Notes / Not Confirmed

- Explicit per-house grouping/boxes inside the dynasty view was not added; houses
  are still identified by the per-node house crest/label and by the divider for
  disconnected lines. This can be a later visual enhancement.
- `dynastyTreeRootIdsFromAllowed` is now unused (superseded by the shared root
  call) and left in place as harmless dead code.
- Verified statically (all modules parse); in-game visual confirmation recommended,
  especially comparing a single-house dynasty (should look near-identical in both
  views) and a multi-house dynasty.
