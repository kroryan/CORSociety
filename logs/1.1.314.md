# Roman Society 1.1.314

## 1.1.314 - "Player appears 3 times" is a NAME COLLISION, not a duplicate

### Real root cause (proven from the code, not assumed)

`characterName` builds the display name as:

```
praenomen + ', ' + houseName(dynasty)
```

The dynasty part (e.g. "Servilius Sidonius") is identical for every member of the
dynasty, and Roman praenomina repeat (~17 exist). So any two DISTINCT characters in
the same dynasty who happen to share the praenomen "Publius" both display as
"Publius, Servilius Sidonius". They are different ids, not the same person.

The mod's graphical tree already de-duplicates by id (`visited` in
`createFamilyTreeBranch`), so the SAME id can never be drawn twice there. Therefore
three "Publius Servilius Sidonius" cards = three different ids (relatives sharing a
praenomen), not the player rendered three times, and not a render/cache bug. This
also explains "missing living members": they are present, but several of them look
identical because they share the name.

### Fixes

- **Name disambiguation in the tree.** Before rendering, the tree collects every
  display name that belongs to more than one distinct id
  (`collectFamilyTreeNameCollisions`). Colliding names are shown with a
  distinguisher (`(b. <year>)`, `(d. <year>)`, or `#<id>` as a last resort) via
  `familyTreeDisplayName`, so distinct relatives stop looking like the same person.
  Nobody is hidden; they are labeled. Applies to House, Dynasty and Known trees.
- **`family` debug dump proves it by id.** `corSocietyDebug("family")` now reports
  `same-name-different-ids [name]: id1, id2, id3` and a `nameCollisions` map, plus
  every member with its id, relations and flags. Use it to confirm the three cards
  are different ids.
- **Ancestor-cycle safety net.** New `isSelfAncestor` + a cycle break in
  `repairFamilyLinkIntegrity`: if a mod-owned character is its own ancestor (corrupt
  data), its parent links are detached so the graph is acyclic. A vanilla family
  tree has no cycle guard, so a cycle there would also render a person repeatedly;
  this removes that possibility.

### Invariants now enforced / checked (mod-owned characters)

1. Not its own father / mother / spouse (cleared if found).
2. `spouseId` never points at a missing character.
3. Spouse links must be mutual (our side cleared otherwise).
4. No ancestor cycle (parent links detached if a character is its own ancestor).
5. Tree render de-duplicates by id (`visited`), so one id is drawn once.
6. Distinct ids that share a display name are disambiguated, never merged.

### About the vanilla tree specifically

If your "Known Family Tree" is the mod's graphical view (title "Known Family
Tree"), the disambiguation above fixes the perceived duplication directly. If it is
the base game's own screen, the duplication there can only come from a data cycle
(now repaired) or the same name collision (a vanilla naming limitation the mod
cannot restyle). Either way the underlying data is now acyclic and the mod views
label distinct same-named people.

### Changed

- Runtime version aligned to `1.1.314` (invalidates the family-tree cache).
- House and Dynasty trees keep their 1.1.311 structure; only name labels gained the
  disambiguator when needed.

### Notes / Not Confirmed

- Please run `corSocietyDebug("family")` and check `same-name-different-ids`. If the
  three "Publius Servilius Sidonius" share ONE id, that would point to a vanilla
  cycle/render issue instead - paste the output and I will target that exact case.
- Reducing future name collisions at generation time (varying praenomina among
  close kin) was intentionally NOT done this release to keep the change small and to
  fix existing saves without regenerating anyone; the display disambiguation covers
  both old and new saves.
- Verified statically (all modules parse); in-game confirmation recommended.
