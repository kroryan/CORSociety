# Roman Society 1.1.308

## 1.1.308 - Family Tree Coherence And Navigation Pass

This release focuses on the graphical family-tree screens (house tree, full
dynasty tree, dead-house tree, and the in-tree node navigation), fixing two
player-visible bugs and the underlying structural causes.

### Fixed

- **House tree could show more members than its dynasty tree.** The house tree is
  documented as "an extracted view from the same complete dynasty tree", but it
  was built by an independent code path (`buildHouseTreeModel`) that did not
  guarantee its members were also part of the dynasty graph. A house could
  therefore display people the dynasty tree omitted, which is impossible if a
  house is a subset of its dynasty. The house tree's allowed-member set is now
  intersected with the dynasty graph (`dynastyTreeAllowedIdMap`) so a house can
  never show members the dynasty omits. The intersection is only applied when the
  dynasty graph actually covers the house (its focus member is present) and falls
  back to the standalone house model otherwise, so it can only ever remove
  out-of-dynasty entries — it can never empty or break the tree. Married-in
  external spouses now render as visual guests in the house tree, consistent with
  the dynasty tree, instead of being treated as full house members.

- **"Back" in family trees did not return to the menu that opened the tree.** The
  back button inferred its destination from the tree `mode` and a `returnTo`
  value that actually described the *origin screen's own parent* (it was passed
  the screen's grandparent navigation context, not a reference to the screen
  itself). Combined with `openHouseFamilyTree`/`openDynastyTree` defaulting
  `returnTo` to `'dynasty'`, and with in-tree node clicks re-using that same
  `returnTo` against a new focus character, "Back" frequently landed on the
  dynasty screen or on whichever character was selected in the tree instead of
  the menu the tree was opened from.

  A tree now carries an explicit `origin` descriptor (`{ method, context }`) that
  records exactly which menu to reopen. The origin is threaded through every tree
  entry point (`openHouseFamilyTree`, `openDynastyTree`, `openFamilyTree`,
  `openDeadHouseFamilyTree`, `openVanillaKnownFamily`, `openVanillaFullFamilyTree`,
  the graphical overlay, and the text fallback) and is preserved unchanged when
  navigating between characters inside the tree, so "Back" always returns to the
  true origin:
  - House tree opened from a house -> returns to that house.
  - Full dynasty tree opened from a house -> returns to that house.
  - Full dynasty tree opened from a dynasty -> returns to that dynasty.
  - Dead-house tree -> returns to that dead house.
  - Relative/person trees -> return to the originating person sheet.

### Changed

- The runtime version is aligned to `1.1.308`. Bumping the version also
  invalidates the per-version family-tree runtime cache, ensuring the rebuilt
  house/dynasty trees are recomputed with the new coherence logic.
- The family-tree "Back" button tooltip now reads "Return to the menu this family
  tree was opened from." instead of the old character-centric wording.

### Notes / Not Confirmed

- Several secondary mechanisms can also make a dynasty view smaller than a house
  view in edge cases (for example a stale `dynasty.houseIds` list, or the 24-root
  cap in `treeRootIdsFromAllowed` dropping branches of very large dynasties).
  These were not reproduced against live save data and are not individually
  patched; the house/dynasty intersection above neutralizes the player-visible
  symptom regardless of which sub-cause triggers. Full unification of the two
  tree builders is recommended but deferred as it needs in-game testing.
- `houseTreeAllowedIdMap` appears to be unused dead code (a parallel house-tree
  filter superseded by `buildHouseTreeModel`). It was left in place this release
  to keep the change set focused; removing it is recommended after confirming no
  dynamic caller exists.
